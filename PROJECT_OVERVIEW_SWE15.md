# FingerMarbleTsWeapp 项目说明（供 Windsurf SWE1.5 使用）

## 1. 项目定位

这是一个基于 **微信小游戏** 运行环境开发的 **2D 弹珠对战/战术捕获小游戏**。

核心玩法可以概括为：

- **玩家与 AI 各控制一颗弹珠**
- **通过拖拽蓄力发射弹珠**
- **弹珠在场景中与边界、障碍物、对方弹珠发生物理碰撞**
- **在回合结束后判断双方是否进入“ 一扎 ”捕获范围**
- **若当前行动方成功将对手纳入捕获距离，则获得胜利**

从代码现状看，它是一个 **单机场景型小游戏原型**，已经具备：

- 基本菜单
- 回合制对抗
- 物理模拟
- 简单积分/经验系统
- 商店与设置页面雏形

但仍处于 **可玩原型 + 功能未完工** 阶段，不适合直接作为成熟上线版本维护。

## 4. 核心架构理解

当前项目是一个 **轻量单体架构**，没有复杂分层，主数据流如下：

```text
game.js
  -> src/game.ts (RetroMarbleGame)
      -> src/eventmanager.ts (输入/菜单/AI/回合)
      -> src/databus.ts (全局数据状态)
      -> src/physics.ts (物理模拟)
      -> src/menu.ts (菜单与商店界面)
```

## 4.1 `game.js`

职责：

- 微信小游戏生命周期入口
- 创建 `RetroMarbleGame` 实例
- 对外挂载 `wx.claimPoints` / `wx.placeBet` / `wx.togglePause`

问题点：

- 文件底部已经直接 `new RetroMarbleGame()`，同时 `wx.onShow` 里又可能再次初始化，生命周期策略不够清晰
- `wx.onHide` 中把 `game = null`，但没有看到事件解绑逻辑，可能导致重复注册触摸事件

## 4.2 `src/game.ts`

职责：

- 游戏主控制器
- 管理游戏状态机
- 初始化物理引擎、菜单、事件系统、技能系统
- 执行 `update` 与 `render`
- 控制玩家/AI 回合与结算时机
- 性能监控和帧率控制

关键状态：

- `MENU`
- `PLAYING`
- `AIMING`
- `MOVING`
- `SETTLING`
- `GAME_OVER`

关键职责：

- `resetGame()`：重置数据、生成玩家/敌人弹珠、随机生成障碍物
- `update(dt)`：推进物理、判断静止、触发 AI 行为、更新技能系统
- `render()`：绘制游戏画面 + 菜单/UI + 技能效果
- `gameLoop()`：帧循环 + 性能监控 + 帧率控制
- `calculateTrajectory()`：计算弹珠预测轨迹
- `getPhysicsEngine()`：获取物理引擎实例

## 4.3 `src/skills.ts` (新增)

职责：

- 技能系统核心管理器
- 技能注册、激活、更新、渲染
- 技能冷却时间和法力消耗管理
- 技能视觉效果渲染

核心技能：

- **冲击波**：推开周围敌人，范围150像素
- **护盾**：临时免疫保护，持续5秒
- **传送**：瞬间移动到指定位置
- **冰冻**：冻结所有敌人3秒
- **加速**：提升移动和发射力量，持续4秒

技术特性：

- 基于时间的冷却系统
- 法力消耗机制
- 活动技能状态管理
- 动态视觉效果渲染

职责：

- 接管微信触摸事件
- 处理菜单点击
- 玩家拖拽发射
- AI 回合行为
- 结算胜负
- 菜单与设置/商店回调桥接

这里实际上承担了 **控制器 + 交互协调器 + 一部分业务规则** 的混合职责，是当前最需要拆分的文件之一。

## 4.4 `src/databus.ts`

职责：

- 全局单例状态管理
- 保存玩家积分、经验、当前弹珠、下注、冷却时间
- 管理物理参数
- 管理游戏对象列表（球、障碍物）
- 提供对象池
- 与微信本地存储交互

特点：

- 是整个项目的“事实状态中心”
- 与 UI、事件、游戏主循环耦合较高
- 同时承担配置、持久化、经济系统、对象创建等多种职责

## 4.5 `src/physics.ts`

职责：

- 维护简化 2D 物理规则
- 更新位移
- 摩擦/速度衰减
- 边界反弹
- 圆-圆碰撞
- 圆-矩形碰撞
- “一扎”距离检测

这是当前玩法的核心基础模块，复杂度不高，但会直接影响“手感”。

## 4.6 `src/menu.ts`

职责：

- 菜单 UI 绘制
- 主菜单、帮助、设置、商店、结算页管理
- 商店道具（弹珠）配置
- 设置项配置
- 按钮布局与事件分发

这里已经包含较多“产品层配置”内容，例如：

- 弹珠商店数据
- 设置项数据
- 各界面标题与文案

比较适合继续演进为“前台 UI 配置中心”。

## 6. 当前主要问题与技术债

以下问题是根据代码和现有 README 综合判断得出的，建议 SWE1.5 优先关注。

## 6.1 已在 README 明确记录的问题

`README.md` 当前记录了以下待处理项：

- 积分保存类型不对
- 回合没有提示
- 不能返回菜单
- 体力和积分没有显示
- 不能分享
- 商店弹珠不能购买
- 体力消耗与显示（TODO）

## 6.2 从代码确认的高优先级问题

### A. 商店购买逻辑存在明确 bug

`src/eventmanager.ts` 中：

- `getMarbleInfo(marbleId)` 使用了 `forEach`
- `forEach` 返回值始终是 `undefined`

这会导致：

- 商店购买回调拿不到正确的弹珠对象
- `marble.cost` / `marble.unlocked` 等逻辑无法正常工作
- README 中“商店弹珠不能购买”与该问题高度对应

这是 **可以明确定位的根因级 bug**。

### B. 菜单积分显示大概率不正确

`MenuSystem.render(state, playerScore = 1000)` 的默认值是 `1000`。

而 `game.ts` 中调用 `this.menu.render(this.menuState)` 时没有传入真实积分，因此：

- 主菜单显示积分可能始终不是实际积分
- 商店可购买判断与界面展示可能不一致

这也是当前体验问题的重要来源。

### C. 物理边界尺寸与真实屏幕尺寸可能不一致

在 `game.ts` 构造函数中：

- `PhysicsEngine` 用 `databus.config.WIDTH/HEIGHT` 初始化

但真实屏幕尺寸是在 `resetGame()` 中才通过 `databus.initConfig(info.width, info.height)` 更新。

意味着：

- 物理引擎边界可能仍保留默认值 `375x667`
- 在非默认尺寸设备上，边界碰撞与渲染尺寸可能不一致

这是 **玩法一致性风险**。

### D. 事件监听可能重复绑定

`EventManager.init()` 中调用：

- `wx.onTouchStart`
- `wx.onTouchMove`
- `wx.onTouchEnd`

但未看到对应的解绑逻辑；同时 `game.js` 里存在重新创建游戏实例的路径。

风险：

- 多次进入前后台后，触摸事件可能叠加
- 出现一次点击触发多次逻辑的情况

### E. 配置存在明显不一致

`databus.ts` 中：

- `config.HAND_SPAN = 20`
- `reset()` 里 `this.handSpan = 120`

这说明：

- 配置与运行值语义不统一
- 玩法数值可能经过临时调试但未归一
- 后续数值平衡难以维护

### F. 分享功能未接通

`game.ts` 中：

- `initWechatShare()` 已引入但被注释掉

说明分享功能处于：

- 设计过
- 但未完成接线或存在兼容性问题

### G. 经济系统持久化不完整

当前只稳定保存：

- 积分
- 当前弹珠

未见可靠持久化：

- 已解锁弹珠状态
- 等级/经验
- 设置项
- 最近领取积分时间

这会导致玩家进度断裂。

### H. `package.json` 元数据仍像模板/类型定义仓库

当前 `package.json` 内容仍是 `minigame-api-typings` 模板风格，并非本项目真实描述。

这意味着：

- 工程元信息不可信
- 自动化工具对项目识别会偏差
- 后续 CI / 文档生成 / 发布说明会受影响

## 7. 当前代码风格与架构特征

SWE1.5 接手时应遵守以下现实约束：

## 7.1 当前代码是“原型驱动开发”风格

表现为：

- 状态和逻辑集中在少数文件中
- UI、业务、数据耦合较高
- 使用较多直接绘制和直接状态改写
- 缺少测试与显式接口边界

因此不建议一上来做大重构，应优先：

- 修明确 bug
- 稳定状态流
- 再做模块化拆分

## 7.2 单例 `DataBus` 是当前事实中心

SWE1.5 修改时要特别注意：

- 很多模块都默认依赖它是全局单例
- 若改接口，通常需要同步修改 `game.ts` / `eventmanager.ts` / `menu.ts`

## 7.3 状态机并未统一定义

目前存在多套状态：

- `game.ts` 的 `GameState`
- `eventmanager.ts` 的 `GameState`
- `DataBus.gameState`
- `MenuState`

这会导致状态语义分裂，是后续重构重点。

建议 SWE1.5 按照以下原则工作：

避免直接把整个项目改成完整 ECS / MVC / 状态机框架；当前更适合：

- 先抽离配置
- 再抽离状态机
- 再抽离 UI 场景
- 最后补测试和工具链

- 玩家拖拽发射
- AI 回合执行
- 回合结算
- 返回菜单
- 商店购买与切换弹珠

### Phase 3：修复高确定性问题

- 优先修 `getMarbleInfo`
- 优先修菜单积分显示
- 优先修生命周期/事件重复绑定

### Phase 4：再做结构优化

- 整理状态与配置
- 拆分大文件
- 增加测试与文档

## 12. 总结

这是一个 **方向明确、玩法核心已经成型，但工程化程度不足的微信小游戏原型**。

其优势在于：

- 核心玩法已经闭环
- 代码规模不大
- 逻辑集中，便于快速定位问题
- 有商店/成长/设置等扩展点

其主要风险在于：

- 状态源不统一
- 生命周期处理粗糙
- 持久化不完整
- 若不先修主线问题，后续新功能会不断叠加技术债

**对 Windsurf SWE1.5 的最佳策略不是立刻大改架构，而是先把“可玩主线 + 真实数据 + 生命周期稳定性”做扎实，再逐步重构。**
